# Skill目录与工具目录合规存放规范
## 文档信息
- 版本：v1.0.0
- 生效日期：2026-02-28
- 审批方：合规团队、产研团队
- 适用范围：全项目所有开发、迭代、运维场景

## 一、定义边界
### 1. Skill目录（/skills/）资源定义
Skill资源指**可复用、场景独立、无核心依赖、支持动态加载/卸载**的业务自定义能力函数，满足以下特征：
- 功能定位：解决特定重复业务场景需求，属于用户/AI自主创建的扩展能力
- 依赖要求：仅允许依赖通用工具、公共库，不允许依赖核心系统模块
- 生命周期：支持动态注册、修改、删除，不影响核心系统稳定性
- 典型场景：自定义数据处理函数、业务专属计算逻辑、特定场景自动化脚本

### 2. 工具目录（/src/tools/）资源定义
工具资源指**系统内置、核心依赖、全局通用、不可随意修改**的基础能力集合，满足以下特征：
- 功能定位：提供系统运行所需的底层基础能力，属于核心系统组成部分
- 依赖要求：被核心系统、agent、技能等多模块依赖，修改会影响全局稳定性
- 生命周期：需经过严格测试、审批后才可修改/新增，不支持动态卸载
- 典型场景：文件操作工具、代码管理工具、web访问工具、系统操作工具

## 二、准入规则
### 1. Skill目录准入规则
| 规则项 | 要求 |
|--------|------|
| 存放路径 | 必须存放在/skills/目录下，禁止嵌套超过2级子目录 |
| 命名规范 | 文件名采用蛇形命名，后缀为.py，名称需清晰表达功能，禁止使用通用名称如utils.py、common.py |
| 内容要求 | 单个文件仅包含一个Skill函数，需添加完整注释说明功能、参数、返回值、使用场景；禁止包含核心系统修改逻辑、恶意代码、敏感信息 |
| 测试要求 | 提交前必须通过单元测试，测试用例存放在/tests/unit/skills/目录下 |
| 权限要求 | 仅支持业务开发、AI自主迭代场景提交，需经过合规校验通过才可写入 |

### 2. 工具目录准入规则
| 规则项 | 要求 |
|--------|------|
| 存放路径 | 必须存放在/src/tools/目录下，按功能分类存放于对应分类文件（如file_tools.py、code_tools.py），禁止随意创建新文件 |
| 命名规范 | 工具函数采用蛇形命名，需包含清晰的功能前缀，统一在/__init__.py中导出才可全局使用 |
| 内容要求 | 必须为全局通用能力，禁止存放业务专属逻辑；需添加完整的注释、参数校验、异常处理逻辑 |
| 测试要求 | 新增/修改工具必须经过全量回归测试，测试覆盖率不低于90% |
| 权限要求 | 仅允许核心开发人员提交，需经过代码评审、合规校验、灰度测试通过才可上线 |

## 三、违规判定标准
### 1. Skill目录违规情形
1. 未存放在/skills/目录下的Skill资源
2. Skill中包含核心系统依赖、修改核心系统代码的逻辑
3. Skill包含恶意代码、敏感信息、未授权访问逻辑
4. 未添加必要注释、无法明确功能的Skill
5. 同一功能重复提交多个Skill
6. 测试不通过仍提交的Skill

### 2. 工具目录违规情形
1. 未存放在/src/tools/目录下的工具资源
2. 工具中包含业务专属逻辑、非通用功能
3. 新增/修改工具未经过测试、审批流程
4. 工具逻辑存在安全漏洞、未做参数校验/异常处理
5. 未在/__init__.py中导出就全局调用的工具函数
6. 随意删除/修改正在被核心系统依赖的工具

## 四、违规处理流程
1. 合规校验系统自动识别违规资源，自动拦截写入操作
2. 已上线的违规资源触发告警，通知提交人3个工作日内整改
3. 逾期未整改的违规资源自动下线，纳入提交人信用记录
4. 造成系统故障的违规资源提交人承担对应责任

## 五、更新规则
本规范每季度评审一次，根据业务需求、系统迭代情况更新，更新需经过合规、产研团队共同确认后生效。
